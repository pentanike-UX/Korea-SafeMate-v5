선택 조건 요약 블록 — 정보 구조 붕괴
칩 구분 없음 · 수정 버튼 3개 분산 · interpretLine 묻힘
explore-journey-summary-bar.tsx
🔴 진단
  • 지역·테마·일정·무드·취향 칩이 모두 동일한 Badge variant="outline"으로 렌더링됨. 사용자가 어떤 기준이 적용됐는지 한눈에 파악 불가.
  • 수정 버튼 3개(조건 수정하기, 일정·인원 수정, 취향 수정)가 우측에 나란히 배치. 어느 것이 주요 액션인지 위계 없음. 동등하게 나열된 버튼은 결국 모두 무시된다.
  • interpretLine(강남·'K-드라마 로맨스 산책'·무드, 혼자·1~2일·균형 기준으로 골랐어요.)이 text-xs border-t mt-1 처리로 묻힘. 이 문장이 사실상 이 블록의 핵심 정보인데 가장 약하게 표시.
  • 레이블 "현재 선택"이 10px uppercase + muted로 처리. 이게 확인 영역인지 필터 패널인지 사용자가 인식하기 어렵다.
🔵 코드 레벨 원인
현재 코드에서 region·theme은 variant="secondary", 나머지 일정·무드·취향 모두 variant="outline"으로 렌더링. secondary와 outline의 시각 차이가 미미해 사실상 동질해 보인다. chipClass도 단일 클래스로 통일되어 있어 카테고리별 컬러 분기 로직 자체가 없다.

수정 버튼은 flex-row flex-wrap justify-end 안에 동일 variant·size·height로 3개 나열. UI 위계 설계가 없다.

interpretLine은 border-t mt-1 pt-2 text-xs로 하단 구석에 숨겨짐. 이 문장을 블록 상단 또는 독립 시각 요소로 격상해야 이 블록이 '선택 확인 영역'으로 기능한다.

공수 S
① interpretLine을 블록 최상단으로 이동하고 시각 격상

현재 border-t mt-1 text-xs로 하단에 묻혀 있는 interpretLine을 블록 제일 위로 올린다. font-size를 최소 text-sm (14px) + font-semibold로 변경. 이 한 줄이 이 블록 존재 이유다.

공수 M
② chipClass를 카테고리별로 분기 — 색상 시스템 적용

현재 단일 chipClass를 카테고리에 따라 분기 처리. 아래 매핑을 기준으로 적용:

region/theme → bg-primary/15 border-primary/30 text-primary (보라) days/party/pace/when → bg-muted/40 border/20 text-muted-fg (중립) sceneMoods → bg-amber-500/10 border-amber-400/25 text-amber-300 (황) styles/tokens → bg-emerald-500/10 border-emerald-400/25 text-emerald-300 (초록)
공수 S
③ 수정 버튼 3개 → "조건 수정 ▾" 단일 드롭다운으로 통합

3개의 동등 버튼을 제거하고 DropdownMenu 1개로 통합. 내부 항목은 기존 3개 액션 그대로. 버튼 크기는 현재 h-9 유지, 위치는 우측 정렬. 버튼 variant는 outline → secondary로 올려 존재감 확보.

공수 S
④ 카테고리 그룹 레이블 추가 (선택 사항, 권장)

칩 앞에 "지역·테마 / 일정·인원 / 무드·취향" 같은 9px uppercase 그룹 레이블을 추가하면 스캔 속도가 올라간다. 칩이 많아질수록 효과가 커진다. 레이블은 muted로 눈에 띄지 않게 처리해 칩 밀도를 해치지 않는다.

1순위 카드 이미지 — 상하단 여백 공백
lg 브레이크포인트에서 이미지 컨테이너 고도 불확정 + object-position 미설정
explore-journey-step-panels.tsx : L.721
🔴 진단
  • 데스크톱(lg)에서 이미지 컨테이너가 aspect-auto + min-h-[300px]. 높이가 우측 콘텐츠 패널 높이에 종속되어 실제 컨테이너가 450~500px으로 늘어남.
  • 이미지는 fill + object-fit:cover이지만 object-position이 기본값 center. 인물 중심이 사진 중앙보다 약간 하단에 있을 경우, 머리 위와 발 아래가 동일하게 잘려 '위아래 공백' 인상을 준다.
  • 사용 중인 이미지가 guardianProfileImageUrls(featured).landscape인데, landscape 이미지를 좁고 긴(320×480) 세로 컨테이너에 넣으면 좌우는 잘리고 상하는 패딩처럼 보이는 영역이 생긴다.
🔵 코드 레벨 원인
문제 클래스:

lg:aspect-auto lg:max-h-none lg:max-w-[320px] lg:min-h-[300px]

aspect-auto는 컨테이너 비율을 포기한다는 의미. 높이는 이제 flexbox 부모가 결정한다. 우측 패널(flex-1 flex-col justify-center) 안에 이름, 티어, 언어, 이유 3줄, CTA 버튼까지 쌓이면 약 420~480px가 된다. 이미지 컨테이너는 이 높이를 그대로 따라가고, landscape 이미지는 320px 너비에서 cover로 맞추면 위아래가 크게 잘려 인물 주변 배경이 노출된다.

FILL_IMAGE_EXPLORE_TOP_PICK 클래스에 object-position이 없으면 Next.js Image의 기본값인 center center가 적용되는데, 이 이미지의 인물 위치가 중앙에서 벗어나 있을 경우 공백이 더 두드러진다.

공수 S
① lg 이미지 컨테이너에 max-h 명시적 제한 추가

현재 lg:max-h-none이 height 제한을 해제. 아래처럼 변경해 컨테이너 최대 고도를 고정한다:

// Before lg:max-h-none lg:min-h-[300px] // After lg:max-h-[420px] lg:min-h-[320px]

420px로 제한하면 우측 콘텐츠가 길어져도 이미지 영역이 과도하게 늘어나지 않는다. 우측 콘텐츠가 넘칠 경우 lg:overflow-y-auto 또는 콘텐츠 gap 줄이기로 대응.

공수 S
② FILL_IMAGE_EXPLORE_TOP_PICK에 object-position: top center 추가

FILL_IMAGE_EXPLORE_TOP_PICK 상수가 정의된 파일을 찾아 아래 클래스를 추가:

// 현재 (추정): object-cover // 변경 후: object-cover object-top

object-top은 Tailwind 기본 유틸리티. 인물 사진은 상체/얼굴이 프레임 위쪽에 위치하는 경우가 많아 object-top이 기본 center보다 안전하다. 특정 이미지에서 어색하면 object-[center_20%] 등 커스텀 포지션으로 미세 조정.

공수 S
③ 이미지 컨테이너 하단에 그라디언트 오버레이 추가 (권장)

컷오프 라인이 자연스럽게 이어지도록 절대 위치 div로 하단 페이드를 추가. 카드 배경색 기준으로 그라디언트 종료:

// 이미지 div 내부에 추가 <div className="absolute inset-x-0 bottom-0 h-1/3 bg-gradient-to-b from-transparent to-card/70 pointer-events-none z-10" />

공백이 실제로 없어지는 건 ①②번 작업이고, 이 그라디언트는 이미지와 콘텐츠 경계를 부드럽게 연결하는 마감 처리다. to-card/70은 현재 카드 배경색 변수를 활용.

공수 M
④ (중기) landscape 대신 portrait 이미지 URL 사용 검토

현재 guardianProfileImageUrls(featured).landscape를 사용 중. 이 카드의 이미지 컨테이너는 320×420px로 세로형이므로 portrait 이미지가 훨씬 잘 맞는다. guardianProfileImageUrls(featured).portrait가 있다면 lg 이상에서 portrait로 전환 검토. 없다면 이미지 등록 시 portrait 버전도 받는 방향으로 데이터 구조 확장.
